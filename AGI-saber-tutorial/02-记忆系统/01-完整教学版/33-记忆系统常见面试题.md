# 33-记忆系统常见面试题

## 1. 一句话结论

面试问记忆系统，不是让你背“短期记忆、长期记忆、图记忆”这几个名词。

真正要讲清楚的是：

```text
一轮对话里，记忆什么时候读、什么时候写、写到哪里、怎么召回、怎么去重、怎么合并、怎么保证最终一致。
```

---

## 2. 建议回答顺序

如果面试官让你介绍记忆系统，按这个顺序讲：

```text
1. 先讲主链路：用户 query 到 LLM 回答的一轮流程
2. 再讲四类记忆：短期、偏好、长期、图记忆
3. 再讲读链路：memPrefix + histMsgs
4. 再讲写链路：规则偏好、LLM 偏好、MemoryWriter
5. 再讲召回：embedding、sim、score、topK
6. 再讲图记忆：seed + Neo4j 邻居扩展
7. 再讲整理：consolidation 去重、合并、过期
8. 最后讲边界和改进：异步、最终一致、分类过滤、补偿重试
```

## 4. 主链路类问题

### Q1：你怎么整体介绍这个记忆系统？

A：

```text
我会按一轮对话讲。用户 query 进入后，系统先写 ShortTermMemory 和 chat_history；然后同步做轻量级规则偏好抽取，同时异步启动 LLM 偏好抽取。回答前会同步召回长期/图记忆，并和偏好一起拼成 memPrefix；短期记忆会通过 ChatHistoryAdapter 转成 histMsgs。LLM 或工具链路生成回答后，assistant answer 写回短期记忆，MemoryWriter 再异步从回答中分类抽取长期记忆。达到 triggerInterval 后，后台 consolidation 做去重、合并、过期清理，并同步 PostgreSQL 和 Neo4j。
```

### Q2：一轮对话里，哪些动作在回答前？

A：

```text
回答前会做：stm.add("user", query)、保存 user chat_history、同步规则偏好抽取、构造 memPrefix、长期/图记忆召回、构造 histMsgs、模式决策。召回是回答前同步执行，所以会影响当前回答。
```

### Q3：一轮对话里，哪些动作在回答后？

A：

```text
回答后会做：stm.add("assistant", answer)、保存 assistant chat_history、MemoryWriter 异步分类写长期记忆、后台 consolidation。回答后写入的长期记忆主要稳定影响后续对话，不保证当前轮用上。
```

### Q4：召回是同步还是异步？为什么？

A：

```text
召回是同步的。buildMemorySystemPrefixWithCtx 会先 llm.embed(query)，再调用 ltm.recall 或 graphMem.recall，拿到相关记忆后拼进 system prompt。如果召回异步，LLM 回答时可能看不到记忆，会影响当前回答质量。异步的是偏好抽取、回答后长期记忆写入、consolidation 和 Neo4j 删除。
```

### Q5：长期记忆到底什么时候写？

A：

```text
长期记忆有两个主要写入时机，不是只有回答后。

第一条是用户 query 进来后，runAsyncPreferenceExtraction(query) 会异步启动。它从用户原话里抽偏好，写 PreferenceMemory 和 preferences 表，同时把偏好转成自然语言事实，生成 embedding 后写入 LongTermMemory / GraphMemory 和 PostgreSQL。这条线程在回答前就启动，但实际完成时间不确定。

第二条是 LLM 回答结束后，memoryWriter.writeAfterReply(query, answer) 会异步启动。它从 assistant answer 里分类抽取值得长期保存的信息，再写入长期记忆、PostgreSQL 和可选 Neo4j。

所以准确说：偏好类长期记忆在用户提问后异步启动写入；回答内容类长期记忆在 LLM 回答后由 MemoryWriter 异步写入。两者都不保证当前轮稳定生效。
```

### Q6：为什么 MemoryWriter 这一路要回答后才写？

A：

```text
这里说的是 MemoryWriter 这一条写入路径，不是所有长期记忆。MemoryWriter 从 assistant answer 中抽取长期记忆，因为回答后 AI 往往已经把用户问题和上下文整理成更清晰的事实。这样能减少直接从混乱 query 中抽错的概率。但代价是 MemoryWriter 写入的新记忆主要影响下一轮。
```

## 5. 短期记忆类问题

### Q7：短期记忆是什么？

A：

```text
短期记忆就是最近聊天记录。代码里是 ShortTermMemory，内部保存 ConversationMessage。用户说一句写 role=user，AI 回答一句写 role=assistant。它解决的是当前对话上下文问题，比如用户说“刚才那个再说一遍”，系统要知道刚才说了什么。
```

### Q8：短期记忆里对象长什么样？

A：

```text
ConversationMessage 大概包含 role、content、timestamp。role 表示 user 或 assistant，content 是消息正文，timestamp 是时间。它不是长期事实，只是最近对话原文。
```

### Q9：短期记忆怎么裁剪？

A：

```text
ShortTermMemory 会按 maxTurns 控制保留轮数。因为一轮对话通常包含 user 和 assistant 两条消息，所以常见限制是 maxTurns * 2。超过后 remove(0) 删除最旧消息。remove(0) 的意思是把列表第一条，也就是最老的一条聊天记录删掉。
```

### Q10：短期记忆和 chat_history 有什么区别？

A：

```text
ShortTermMemory 是 JVM 内存里的最近上下文，用于当前 LLM messages。chat_history 是数据库聊天历史，用于持久化。短期记忆更像桌面便签，chat_history 更像档案柜。
```

### Q11：histMsgs 是什么？

A：

```text
histMsgs 是 ChatHistoryAdapter 从 ShortTermMemory 转出来的 LLM messages。它里面是最近几轮对话，格式类似 {"role":"user","content":"..."}。它解决的是 LLM 要看到最近聊天上下文的问题。
```

## 6. 偏好记忆类问题

### Q12：偏好记忆是什么？

A：

```text
偏好记忆保存用户稳定习惯，比如姓名、默认城市、回答风格、喜欢的解释方式。代码里是 PreferenceMemory，通常是 key-value 形式，例如 "回答偏好" -> "喜欢 Java 逐行解释"。
```

### Q13：偏好抽取有几条路径？

A：

```text
两条。第一条是 pref.extractAndSave(query)，同步轻量级规则抽取，不调用大模型，速度快但能力弱。第二条是 runAsyncPreferenceExtraction(query)，异步调用 LLM 抽取偏好，理解更强但完成时间不稳定。
```

### Q14：轻量级规则什么时候用？

A：

```text
每轮 query 进来后都会同步跑规则偏好抽取。它适合识别“我叫”“我喜欢”“我爱”这类明显表达。因为不调用 LLM，所以快，能尽量在当前轮生效。
```

### Q15：LLM 偏好抽取什么时候用？

A：

```text
每轮 query 进来后也会启动 runAsyncPreferenceExtraction。它在新线程里调用 llm.extractPreferences(query)，抽出 Map<String,String> 后写入 PreferenceMemory、preferences 表，并转成长期记忆。它不保证当前轮完成，主要影响后续轮次。
```

### Q16：规则偏好和 LLM 偏好冲突怎么办？

A：

```text
当前代码没有复杂冲突仲裁机制，通常是后写入覆盖或补充前面的结果。面试可以说这是一个可改进点：可以给偏好加 source、confidence、updatedAt，规则抽取低置信度，LLM 抽取高置信度，再按置信度和时间决定是否覆盖。
```

### Q17：偏好会不会进入长期记忆？

A：

```text
会。runAsyncPreferenceExtraction 会把 LLM 抽出的偏好转成自然语言事实，例如“用户回答偏好: 喜欢 Java 逐行解释”，生成 embedding 后写入 LongTermMemory 或 GraphMemory。MemoryWriter 也会把 category=preference 的内容写入长期记忆。
```

### Q18：PreferenceMemory 和长期记忆里的 preference 有什么区别？

A：

```text
PreferenceMemory 是结构化 KV，适合直接拼到【用户偏好】里。长期记忆里的 category=preference 是自然语言事实，有 embedding，可以参与语义召回和图关系。一个偏好可以同时存在于两边。
```

## 7. 长期记忆写入类问题

### Q19：长期记忆是什么？

A：

```text
长期记忆是跨轮对话保存的稳定事实。代码里是 MemoryItem，包含 id、content、importance、embedding、category、tags、slotHint、createdAt、lastAccessed、score 等字段。
```

### Q20：MemoryItem 里最重要的字段是什么？

A：

```text
id 用于内存、PostgreSQL、Neo4j 对齐；content 是自然语言记忆内容；importance 是写入时的重要性；embedding 用于语义召回和去重；category/tags 用于分类召回；score 是召回时临时计算的排序分。
```

### Q21：长期记忆怎么加入？

A：

```text
主要有两条写入路径。第一条是异步 LLM 偏好抽取，把偏好转成自然语言事实写入长期记忆。第二条是回答后的 MemoryWriter，从 answer 里分类抽取值得长期保存的信息，生成 embedding 后调用 ltm.storeClassified 或 graphMem.storeClassified，新增成功后保存 PostgreSQL，并同步数据库 ID。
```

### Q22：添加记忆时会不会判断重复？

A：

```text
会。storeClassified 在新增前会拿新 embedding 和已有 MemoryItem.embedding 计算 cosine 相似度。如果 sim >= consolidationCfg.dedupThreshold，就认为重复，不新增，只更新已有记忆的 importance、category、tags、slotHint、lastAccessed 等信息。
```

### Q23：为什么添加时已经去重，还要 consolidation？

A：

```text
添加时去重只比较“当前新记忆”和已有记忆，而且依赖 embedding 可用。consolidation 是定期全量整理，会对所有记忆做衰减、两两比较、重复删除、相似合并、过期删除，并同步 DB 和图。添加时去重是入口防抖，consolidation 是后台清理。
```

### Q24：importance 是怎么来的？

A：

```text
MemoryWriter 根据 category 映射 importance：identity=0.9，policy=0.8，preference=0.7，tool_failure=0.6，general=0.5。runAsyncPreferenceExtraction 写偏好长期记忆时使用 0.8。importance 是写入时确定的，不是召回时算的。
```

### Q25：embedding 是对什么内容做的？

A：

```text
写入长期记忆时，embedding 是对记忆 content 做的，不是对原始 query 做的。例如 content="用户偏好 Java 代码逐行解释"，就对这句话做 llm.embed(content)。召回时才对 query 做 embedding，用 queryEmbedding 和 memory embedding 比较。
```

### Q26：syncLastItemPGID 是干什么的？

A：

```text
长期记忆先写入内存，此时 MemoryItem.id 可能是内存里的临时自增 ID。保存 PostgreSQL 后，数据库会返回真正的自增 id。syncLastItemPGID(pgId) 会把内存里最后一条 MemoryItem.id 改成 pgId，保证内存 ID、PostgreSQL id、Neo4j mem_id 对齐。
```

## 8. 长期记忆召回类问题

### Q27：长期记忆怎么召回？

A：

```text
回答前系统会先对 query 做 embedding，然后遍历长期记忆，计算 queryEmbedding 和 MemoryItem.embedding 的 cosine 相似度，再用 score = sim * 0.7 + importance * 0.3 得到最终排序分，超过阈值的候选按 score 排序取 topK。
```

### Q28：score 和 sim 有什么区别？

A：

```text
sim 是 query 和记忆内容的语义相似度。score 是召回排序综合分，既看 sim，也看 importance。公式是 score = sim * 0.7 + importance * 0.3。sim 只表示像不像，score 表示最终值不值得召回。
```

### Q29：score 是什么时候来的？

A：

```text
score 是召回时临时计算出来的。它不是写入长期记忆时固定保存的字段。每次 query 不同，sim 不同，所以 score 也会不同。
```

### Q30：分类召回怎么做？

A：

```text
改进后的主链路会构造 LongTermFilter，然后调用 recallByFilter。recallByFilter 先按 category、requiredTags、maxAgeHours 过滤候选，再计算 sim 和 score，最后排序取 topK。这样能让 MemoryWriter 写入的 category/tags 真正参与召回。
```

### Q31：category 有哪几种？

A：

```text
当前 Java 版 MemoryWriter normalizeCategory 固定支持五类：identity、preference、tool_failure、policy、general。未知 category 会归一化成 general。
```

### Q32：tags 有哪几种？

A：

```text
tags 不是固定枚举。它由 LLM 输出，例如 ["Java","面试"]、["城市","天气"]、["回答风格"]。代码只过滤空字符串，不校验固定集合。所以 category 是固定大类，tags 是自由标签。
```

### Q33：过滤越细越好吗？

A：

```text
不是。过滤太粗会召回无关记忆，过滤太细会漏掉有用记忆。尤其 requiredTags 当前是 AND 关系，如果塞很多 tag，一条记忆必须同时包含所有 tag 才能通过。更合理的是 category 做粗过滤，tags 只在明确场景做弱约束或少量强过滤，最后交给 embedding 和 score 排序。
```

### Q34：多意图 query 怎么召回更好？

A：

```text
多意图 query 不适合整句只召回一次。例如“复习图记忆，顺便查上海天气”包含学习和天气两个意图。更好的做法是先拆成子 query，分别召回，再按 MemoryItem.id 合并去重，最后按 score 排序。
```

## 9. 图记忆类问题

### Q35：图记忆是什么？

A：

```text
图记忆是在长期记忆上叠加 Neo4j 图层。长期记忆负责存 MemoryItem 和 embedding，图记忆负责存 Memory 节点和关系边，例如 FOLLOWS、SIMILAR_TO、CAUSES、BELONGS_TO。它解决的是记忆之间关系和邻居扩展问题。
```

### Q36：图记忆怎么写入？

A：

```text
GraphMemory.storeClassified 先调用 ltm.storeClassified 写长期记忆。如果新增成功，就异步 upsert Neo4j Memory 节点；如果 prevId 存在，建立 FOLLOWS 边；然后对最近一批旧记忆算 embedding 相似度，超过 simThreshold 就建立 SIMILAR_TO 边。
```

### Q37：FOLLOWS 边什么时候创建？

A：

```text
新增图记忆时，如果 prevId >= 0，就创建 prevId -> newId 的 FOLLOWS 边，权重 1.0。它表示写入顺序，不表示因果。
```

### Q38：SIMILAR_TO 边什么时候创建？

A：

```text
新增记忆后，GraphMemory 会拿新记忆和最近最多约 50 条旧记忆比较 embedding cosine 相似度。如果 sim >= simThreshold，就创建 oldId -> newId 的 SIMILAR_TO 边，边权重是 sim。
```

### Q39：图记忆怎么召回？

A：

```text
GraphMemory.recall 先调用 ltm.recall 召回 seed 记忆，再把 seedIds 传给 KGStore.expandMemoryNeighbors，在 Neo4j 里扩展 1 跳邻居。扩展出的 ID 如果能在内存 LongTermMemory.items 找到，就加入结果并设置一个较低 score，然后合并排序取 topK。
```

### Q40：图记忆召回的 score 怎么来？

A：

```text
seed 记忆的 score 来自长期记忆召回公式：score = sim * 0.7 + importance * 0.3。图扩展出来的邻居在当前 Java 代码里会被设置为固定分数 0.45，然后和 seed 一起排序。
```

### Q41：图记忆有 BELONGS_TO 吗？

A：

```text
代码注释和 KGStore 支持 BELONGS_TO 这种边类型，但当前 GraphMemory.storeClassified 自动创建的是 FOLLOWS 和 SIMILAR_TO。BELONGS_TO 当前不是自动写入的主路径。
```

## 10. MemoryWriter 分类写入类问题

### Q42：MemoryWriter 是干什么的？

A：

```text
MemoryWriter 是回复后的长期记忆写入器。它从 assistant answer 中识别值得长期保存的信息，要求 LLM 输出 JSON items，每个 item 包含 category、content、tags，然后按类别写入 PreferenceMemory、LongTermMemory、PostgreSQL 和可选 Neo4j。
```

### Q43：MemoryWriter 为什么从 answer 抽，不直接从 query 抽？

A：

```text
query 可能口语化、多意图、不完整；answer 通常经过 LLM 整理，事实表达更规范。基于 answer 抽长期记忆，能得到更适合保存的自然语言事实。但如果 answer 自身有错误，也可能把错误写入长期记忆，这是风险点。
```

### Q44：MemoryWriter 的分类有哪些？

A：

```text
identity 表示身份信息；preference 表示偏好；tool_failure 表示工具调用失败经验；policy 表示用户硬性约束；general 表示兜底事实。未知分类会变成 general。
```

### Q45：identity / preference 为什么还要写 PreferenceMemory？

A：

```text
因为 identity 和 preference 既适合做长期语义召回，也适合直接作为结构化用户偏好进入 system prompt。所以 MemoryWriter 对这两类会尝试 guessPrefKey，然后写 PreferenceMemory 和 preferences 表，同时也写长期记忆。
```

### Q46：slotHint 是什么？

A：

```text
slotHint 是给 prompt 上下文装配用的提示。identity/preference 映射 Profile，policy 映射 Constraints，tool_failure 映射 ToolState，general 没有 slotHint。它帮助后续把记忆放进更合适的上下文槽位。
```

## 11. consolidation 整理类问题

### Q47：consolidation 是干什么的？

A：

```text
consolidation 是长期记忆后台整理。它不是召回，而是维护记忆库质量：降低旧记忆 importance，删除重复记忆，合并相似记忆，清理过期且不重要的记忆，并同步数据库和图数据库。
```

### Q48：什么时候触发 consolidation？

A：

```text
needConsolidation 需要三个条件：consolidationCfg 不为空，triggerInterval > 0，storeCount >= triggerInterval。也就是开启整理配置，并且新增记忆数量达到触发间隔。
```

### Q49：consolidation 的三阶段是什么？

A：

```text
第一阶段 Decay：按记忆年龄衰减 importance。第二阶段 Dedup + Merge：两两比较，超过 dedupThreshold 的删除重复，超过 similarityThreshold 的合并。第三阶段 Expire：超过 ttlDays 且 importance 低于 minImportance 的记忆删除。
```

### Q50：两两比较在哪里做？

A：

```text
在 LongTermMemory.consolidate 的 Phase 2。外层 for i 遍历每条记忆，内层 for j 从 i+1 开始比较后面的记忆，调用 itemSimilarity(items.get(i), items.get(j)) 计算相似度。
```

### Q51：mergeItems 怎么合并 content？

A：

```text
先选 importance 更高的作为 base。若 base.content 和 other.content 互不包含，就用分号拼接；如果一个内容包含另一个，就保留更长或更完整的内容。这样避免重复文本反复拼接。
```

### Q52：embedding 怎么合并？

A：

```text
如果两条记忆 embedding 维度一致，会按 importance 做加权平均。公式是 merged[i] = (base[i] * wA + other[i] * wB) / (wA + wB)。这个叫加权平均，也可以叫 Weighted Vector Average。
```

### Q53：过期删除条件是什么？

A：

```text
记忆年龄 days > ttlDays，并且 importance < minImportance，才会过期删除。也就是说旧但重要的记忆不会轻易删，旧且不重要的记忆才删。
```

## 12. 一致性和数据库类问题

### Q54：合并后怎么保证内存、PostgreSQL、Neo4j 一致？

A：

```text
当前 Java 版靠统一 ID 和 ConsolidationResult 做最终一致。ltm.consolidate 先整理内存 items，并返回 result。result.deleteFromDB 记录要删除的数据库 ID，result.updateInDB 记录合并后要更新的 MemoryItem。syncConsolidationToDB 根据 result 删除/更新 PostgreSQL。图模式下 graphAwareConsolidate 还会根据 result.deleteFromDB 删除 Neo4j Memory 节点。
```

### Q55：这是强一致吗？

A：

```text
不是强一致，是最终一致。consolidation 在后台线程执行，Neo4j 删除也是异步线程，所以短时间内可能出现内存已经整理完，但 PostgreSQL 或 Neo4j 还没同步完成。
```

### Q56：图模式中心度保护是怎么做的？

A：

```text
graphAwareConsolidate 会先调用 ltm.consolidate 得到 deleteFromDB，然后用 kg.getHighCentralityMemoryIds(deleteFromDB, 3) 查 Neo4j 中入度较高的候选节点。入度 >= 3 的节点会从 deleteFromDB 过滤掉，避免从 DB 和 Neo4j 删除。
```

### Q57：图保护有什么边界？

A：

```text
当前 Java 版是先 ltm.consolidate 改内存，再做图中心度保护。所以图保护主要保护 PostgreSQL 和 Neo4j 不删除，不能严格保证被保护节点仍然留在当前 JVM 内存 items 中。更好的做法是 plan/apply 两阶段，先查图保护，再统一修改内存、DB、Neo4j。
```

### Q58：如果 DB 更新成功但 Neo4j 删除失败怎么办？

A：

```text
当前代码主要记录 warn 日志，没有完整补偿事务。面试可以说这是改进点：引入 outbox 或补偿任务表，记录待同步的 graph delete/update 操作，失败后定时重试，保证最终一致。
```

### Q59：restoreFromDB 是干什么的？

A：

```text
服务启动时从 PostgreSQL 恢复偏好和长期记忆到内存。长期记忆表中的 id、content、importance、embedding、category、tags、slotHint 会恢复成 MemoryItem，保证重启后内存不是空的。
```

## 13. 工具链路相关问题

### Q60：记忆怎么进入工具调用链路？

A：

```text
UnifiedAgentService 会把 memPrefix 和 histMsgs 传给 toolHandler.run 或 reactLoop.runStream。memPrefix 里有用户偏好和相关记忆，histMsgs 里有最近对话。工具链路可以利用这些上下文生成工具参数或规划工具调用。
```

### Q61：PreferenceFiller 和偏好记忆有什么关系？

A：

```text
PreferenceFiller 可以用 PreferenceMemory 里的结构化偏好补工具参数。例如用户默认城市是上海，天气工具需要 city 参数，如果当前 query 没写城市，就可以从偏好里补 city=上海。
```

### Q62：tool_failure 为什么要存长期记忆？

A：

```text
tool_failure 用来记录工具调用失败经验，例如参数错误、工具不可用、网络失败。后续 tool/react 模式召回这些记忆，可以避免重复犯同样错误，或者提前调整参数和工具选择。
```

## 14. 设计取舍和改进类问题

### Q63：这个记忆系统最大的风险是什么？

A：

```text
第一，异步写入导致新记忆不保证当前轮生效。第二，LLM 抽取可能抽错，错误事实会进入长期记忆。第三，图和 DB 是最终一致，失败时需要补偿机制。第四，整句 embedding 对多意图 query 不够准确，需要 query 拆分和分类召回。
```

### Q64：如果要提高召回准确率，你怎么改？

A：

```text
我会做四点：第一，引入 query 拆分，把多意图问题拆成子 query 分别召回；第二，用 LongTermFilter 做 category 粗过滤；第三，tags 做弱约束或少量强过滤，避免 AND 过滤太死；第四，把向量召回、重要性、时间新鲜度、图邻居权重综合排序。
```

### Q65：如果要提高写入准确率，你怎么改？

A：

```text
我会给 MemoryWriter 增加更严格的 JSON schema、置信度 confidence、来源 source、人工可解释字段，并避免直接相信 answer。对于 identity/policy 这类高风险记忆，可以要求 query 和 answer 双重证据，或者设置更高写入阈值。
```

### Q66：如果要增强一致性，你怎么改？

A：

```text
我会把 consolidation 改成 plan/apply 两阶段。plan 阶段只计算删除、合并、图保护，不改内存；apply 阶段按最终计划更新内存、PostgreSQL 和 Neo4j。再加 outbox 补偿表，DB 或 Neo4j 同步失败时可以重试。
```

### Q67：如果要优化性能，你怎么改？

A：

```text
长期记忆多了以后，遍历所有 MemoryItem 算相似度会慢。我会引入向量数据库或 ANN 索引，例如 pgvector、Milvus、Faiss，把 topK 向量检索下推到索引层。同时 consolidation 的两两比较是 O(n^2)，可以按 category/tags 分桶后再比较。
```

### Q68：如果面试官问“这个系统做得不准怎么办”怎么答？

A：

```text
我会承认当前基础实现有局限：整句 embedding 对多意图不准，LLM 抽取可能误写，异步最终一致有延迟。然后讲改进方案：query 拆分、分类召回、置信度写入、记忆过期和合并、补偿重试、向量索引和评测集。重点是说明我知道问题在哪里，也知道怎么工程化改。
```

## 15. 最容易说错的点

```text
1. 召回不是异步，召回是回答前同步。
2. 异步的是偏好 LLM 抽取、MemoryWriter、consolidation、Neo4j 删除。
3. histMsgs 是短期聊天历史，不是长期记忆。
4. memPrefix 是偏好 + 相关长期/图记忆，不是聊天历史。
5. importance 是写入时确定，score 是召回时计算。
6. score 不是数据库固定字段，而是每次 query 临时算。
7. category 是固定 5 类，tags 是自由标签。
8. requiredTags 当前是 AND 关系，不是 OR。
9. FOLLOWS 是顺序边，不是因果边。
10. SIMILAR_TO 是相似边，来自 embedding 相似度超过 simThreshold。
11. BELONGS_TO 当前不是自动写入主路径。
12. syncLastItemPGID 同步的是 ID，不是内容。
13. consolidation 不是召回，是后台整理。
14. 添加时去重不能替代 consolidation。
15. 内存、PostgreSQL、Neo4j 是最终一致，不是强一致。
```

## 16. 一段完整面试回答

可以这样开场：

```text
我这个记忆系统可以按一轮对话来讲。用户 query 进来后，系统先写入 ShortTermMemory 和 chat_history；同时做两类偏好抽取，一类是同步轻量规则，适合“我叫”“我喜欢”这种明显表达，另一类是异步 LLM 偏好抽取，理解能力更强但主要影响后续轮次。

回答前，系统会同步调用 buildMemorySystemPrefixWithCtx，对 query 做 embedding，然后从 LongTermMemory 或 GraphMemory 召回相关记忆。召回结果和 PreferenceMemory 里的偏好一起拼成 memPrefix，短期聊天历史通过 ChatHistoryAdapter 转成 histMsgs。memPrefix 进入 system prompt，histMsgs 进入 messages。

回答后，assistant answer 写回短期记忆和 chat_history。MemoryWriter 会在后台从 answer 中分类抽取长期记忆，类别包括 identity、preference、policy、tool_failure、general，不同类别映射不同 importance。写入时会生成 embedding，做重复判断，新增成功后保存 PostgreSQL；如果启用图记忆，还会写 Neo4j Memory 节点，并建立 FOLLOWS 和 SIMILAR_TO 边。

长期记忆召回时会计算 sim 和 score，score = sim * 0.7 + importance * 0.3。图记忆召回是在长期记忆 seed 的基础上扩展 Neo4j 邻居。后台 consolidation 会在 storeCount 达到 triggerInterval 后触发，做 importance 衰减、去重、相似合并和过期清理。合并后通过 ConsolidationResult 同步 PostgreSQL 和 Neo4j。当前实现是最终一致，不是强一致；如果要增强，我会做 plan/apply 两阶段和补偿重试。
```
