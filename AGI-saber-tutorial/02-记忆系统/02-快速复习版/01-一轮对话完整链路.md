# 01-一轮对话完整链路

## 1. 一句话结论

一轮对话的真实顺序是：

```text
先写 user 短期记忆
再抽偏好
再同步召回长期/图记忆
再调用 LLM / 工具
最后写 assistant 短期记忆和异步长期记忆
```

---

## 2. 真实源码顺序

核心入口：

```text
UnifiedAgentService.processInternal
```

简化版顺序：

```java
stm.add("user", query);
infra.saveChatHistory("user", query);

runAsyncPreferenceExtraction(query);
pref.extractAndSave(query);

String memPrefix = buildMemorySystemPrefixWithCtx(query);
List<Map<String, String>> histMsgs = ChatHistoryAdapter.buildHistory(stm, query);

String mode = ChatRouter.decideMode(...);

switch (mode) {
    case "react" -> reactLoop.runStream(..., memPrefix, histMsgs, ...);
    case "tool" -> toolHandler.run(..., memPrefix, histMsgs);
    case "rag" -> rag.query(query);
    default -> llm.chat(systemPrompt, histMsgs);
}

stm.add("assistant", resp.getAnswer());
infra.saveChatHistory("assistant", resp.getAnswer());

memoryWriter.writeAfterReply(query, resp.getAnswer());

new Thread(() -> {
    if (graphMem != null && graphMem.needConsolidation()) {
        syncConsolidationToDB(graphMem.graphAwareConsolidate());
    } else if (ltm.needConsolidation()) {
        syncConsolidationToDB(ltm.consolidate());
    }
}).start();
```

---

## 3. 当前轮一定会影响回答的部分

```text
stm.add("user", query)
pref.extractAndSave(query)
buildMemorySystemPrefixWithCtx(query)
ltm.recall / graphMem.recall
ChatHistoryAdapter.buildHistory
```

原因：

```text
这些都在 LLM 回答前同步执行。
```

---

## 4. 当前轮不保证生效的部分

```text
runAsyncPreferenceExtraction(query)
memoryWriter.writeAfterReply(query, answer)
consolidation
Neo4j 删除
```

原因：

```text
这些都在新线程里执行，主要影响后续轮次。
```

---

## 5. 真实例子

用户说：

```text
我叫小李，我喜欢 Java 逐行解释。帮我复习图记忆怎么召回。
```

执行过程：

```text
1. 短期记忆写入 user 原话。
2. 规则偏好可能同步抽出：姓名=小李，喜好=Java 逐行解释。
3. LLM 偏好抽取异步启动，不保证当前轮完成。
4. 回答前同步召回已有长期/图记忆。
5. memPrefix = 偏好 + 相关记忆。
6. histMsgs = 最近聊天历史。
7. LLM 生成回答。
8. assistant 回答写入短期记忆。
9. MemoryWriter 从 answer 里异步抽长期记忆。
10. 达到 triggerInterval 后后台整理。
```

---

## 6. 面试怎么说

```text
这条链路里，读记忆发生在回答前，写新长期记忆主要发生在后台。这样保证当前回答能拿到已有记忆，同时避免长期记忆写入、LLM 抽取、Neo4j 操作阻塞用户响应。
```
