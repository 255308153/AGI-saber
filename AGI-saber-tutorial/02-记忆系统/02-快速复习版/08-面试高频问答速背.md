# 08-面试高频问答速背

## Q1：这个记忆系统整体怎么讲？

A：

```text
按一轮对话讲。用户 query 先写短期记忆，同时抽偏好；回答前同步召回长期/图记忆，拼成 memPrefix，短期历史转成 histMsgs；回答后写 assistant 短期记忆，MemoryWriter 异步写长期记忆，后台 consolidation 做整理。
```

## Q2：短期记忆是什么？

A：

```text
短期记忆是最近聊天记录，保存 ConversationMessage，包括 role、content、timestamp。它通过 histMsgs 进入 LLM messages。
```

## Q3：偏好记忆是什么？

A：

```text
偏好记忆是用户稳定习惯，比如姓名、默认城市、回答风格。它通常是 KV，进入 memPrefix 的【用户偏好】。
```

## Q4：长期记忆什么时候写？

A：

```text
两条路径。query 后 runAsyncPreferenceExtraction 异步抽偏好并写长期记忆；answer 后 MemoryWriter 异步从回答里分类抽取长期记忆。两者都不保证当前轮稳定生效。
```

## Q5：召回是同步还是异步？

A：

```text
召回是同步。回答前 buildMemorySystemPrefixWithCtx 会先 embed query，再调用 ltm.recall 或 graphMem.recall。召回结果进入 system prompt，所以必须在回答前完成。
```

## Q6：score 怎么算？

A：

```text
score = sim * 0.7 + importance * 0.3。sim 是 query 和记忆 embedding 的相似度，importance 是记忆写入时的重要性，score 是召回时临时算的排序分。
```

## Q7：importance 怎么来？

A：

```text
MemoryWriter 按 category 映射：identity=0.9，policy=0.8，preference=0.7，tool_failure=0.6，general=0.5。偏好异步抽取写长期记忆时使用 0.8。
```

## Q8：category 和 tags 区别？

A：

```text
category 是固定五类：identity、preference、tool_failure、policy、general。tags 是 LLM 输出的自由标签，比如 Java、面试、天气、回答风格。
```

## Q9：过滤越细越好吗？

A：

```text
不是。过滤太细会漏掉有用记忆。requiredTags 当前是 AND 关系，所以多意图 query 不要塞一堆 tag，应该拆成子 query 分别召回。
```

## Q10：图记忆怎么召回？

A：

```text
先用长期记忆召回 seed，再拿 seedIds 去 Neo4j 扩展一跳邻居，把邻居 ID 对应的 MemoryItem 加入结果，最后合并排序取 topK。
```

## Q11：FOLLOWS 是什么？

A：

```text
FOLLOWS 是顺序边。新增图记忆时，如果有上一条记忆 prevId，就建立 prevId -> newId。它表示写入顺序，不表示因果。
```

## Q12：SIMILAR_TO 什么时候创建？

A：

```text
新增记忆后，拿新记忆和最近一批旧记忆做 embedding 相似度比较。如果 sim >= simThreshold，就创建 SIMILAR_TO 边，权重是 sim。
```

## Q13：为什么添加时去重后还要 consolidation？

A：

```text
添加时去重只处理当前新记忆和已有记忆。consolidation 是定期全量整理，会做 importance 衰减、两两比较、重复删除、相似合并、过期删除和 DB/图同步。
```

## Q14：合并时 embedding 怎么处理？

A：

```text
用 importance 加权平均：merged[i] = (a[i] * wA + b[i] * wB) / (wA + wB)。这叫 Weighted Vector Average。
```

## Q15：内存、PostgreSQL、Neo4j 怎么一致？

A：

```text
靠统一 ID 和 ConsolidationResult 做最终一致。内存先整理，result 记录 deleteFromDB 和 updateInDB，PostgreSQL 根据 result 删除/更新，Neo4j 根据 deleteFromDB 删除节点。
```

## Q16：这是强一致吗？

A：

```text
不是，是最终一致。consolidation 和 Neo4j 删除都是后台异步，短时间内可能内存已更新但 DB/Neo4j 还没同步完。
```

## Q17：图保护有什么问题？

A：

```text
当前 graphAwareConsolidate 先 ltm.consolidate 改内存，再查 Neo4j 中心度保护。所以图保护主要保护 DB 和 Neo4j，不严格保护当前 JVM 内存 items。
```

## Q18：怎么改进召回？

A：

```text
做 query 拆分、分类粗过滤、tags 弱约束、向量 + importance + 新鲜度 + 图权重综合排序。长期记忆多时引入向量索引或 pgvector。
```

## Q19：怎么改进一致性？

A：

```text
改成 plan/apply 两阶段，先生成删除/合并计划并做图保护，再统一更新内存、PostgreSQL、Neo4j。失败时用 outbox 或补偿任务重试。
```

## Q20：最重要的坑是什么？

A：

```text
召回不是异步；长期记忆不是只有回答后才写；importance 不是 score；FOLLOWS 不是因果；tags 不是固定枚举；DB/图/内存不是强一致。
```
