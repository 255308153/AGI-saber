# 23 主链路简历说法

## 30 秒电梯演讲

> 这个系统是一个 AI Agent 的主链路，实现了一个请求从接收、路由、多模式处理、记忆存取到响应的完整流程。四层记忆架构（STM/Pref/LTM/Graph）配合四种路由模式（chat/tool/react/rag）和一个 DAG 并发执行引擎，核心是给 LLM 提供最优的上下文，同时把有用的信息持久化。

## 2 分钟项目概述

适用场景：面试自我介绍中最开始讲项目。

> 我做的是一个 Java 实现的 AI Agent 主链路。它就像一个 AI 大脑的核心调度系统：收到用户问题后，先存到短期记忆，抽取用户偏好，然后根据关键词判断走 chat/tool/react/rag 四种模式之一。
>
> 如果是工具调用，它支持单工具和多工具两种方式。单工具直接调一个 API 然后合成回答。多工具更复杂——系统会先用 LLM 规划一个 DAG（有向无环图），然后把无依赖的节点并行执行，支持竞速机制（多个同功能工具里取最快结果），最后再让 LLM 合成回答。
>
> 回答生成后写入短期记忆和聊天历史，同时后台异步 MemoryWriter 会调用 LLM 从回答里分类提取值得长期记忆的信息，写入长期记忆和图记忆，定期 consolidation 做衰减、去重、合并、过期清理。
>
> 整个系统大概有 10 个步骤，核心调度在 UnifiedAgentService 里。没有用分布式组件，原型阶段追求简单、降级友好。

## 5 分钟深度讲解

适用场景：面试官表现出兴趣，想深入。

### 第一层：请求处理流程

从 `AgentController` 接收到 `ChatApplicationService.processStream`，再到 `UnifiedAgentService.processInternal`：

```text
① stm.add("user", query) + saveChatHistory
② runAsyncPreferenceExtraction(query)
③ buildMemorySystemPrefixWithCtx(query)
④ ChatHistoryAdapter.buildHistory
⑤ ChatRouter.decideMode
⑥ 模式执行（chat/tool/react/rag）
⑦ stm.add("assistant", answer) + saveChatHistory
⑧ 填充响应元信息
⑨ MemoryWriter.writeAfterReply（异步）
⑩ 返回 ChatResponse
```

10 步中第 ⑤ 步是路由决策，第 ⑥ 步是模式执行（最复杂），第 ⑨ 步是异步记忆写入。

### 第二层：路由决策

`ChatRouter.decideMode` 使用纯关键词匹配。不用 LLM，不消耗 token。

| 模式 | 触发条件 | 执行方式 |
|---|---|---|
| chat | 无关键词 | 直接 LLM对话 |
| tool | 1 类任务关键词 | 单工具+LLM合成 |
| react | >=2 类任务关键词 | 规划+并行执行+LLM合成 |
| rag | 非工具且知识库已加载 | 召回+LLM合成 |

面试时强调：**4 种模式的优先级是固定的，React 优先于 Tool 优先于 RAG 优先于 Chat。** 这个顺序保证了"能调工具先调工具，调不了再聊天"。

### 第三层：tool 模式

`ToolModeHandler` 走 5 步：

```text
selectTools → getExecute → apply → LLM 合成 → 保存 steps
```

关键设计：

- `selectTools` 用 query 匹配工具关键词，选匹配度最高的。
- `tool.getExecute()` 返回 `ExecuteFunction`，这是函数式接口。
- `execute.apply(args)` 执行调用，返回结果。
- LLM 用工具结果+原问题合成回答。

### 第四层：ReAct 模式

ReAct 模式是系统的核心亮点。

```text
Planner.planGraph → GraphRuntime.execute → LLM synthesis
```

Planner 把 query + 工具列表发给 LLM，LLM 规划 DAG。Planner 有 **3 层 JSON 解析降级**：

| 降级层级 | 策略 |
|---|---|
| 第 1 层 | 直接解析 LLM 原始 JSON |
| 第 2 层 | 清洗后解析（去掉 Markdown 标记） |
| 第 3 层 | 规则规划（每个工具独立节点） |
| 兜底 | 降级到 chat 模式 |

GraphRuntime 用 **拓扑排序** 把 DAG 分层，层内并行执行。

并发控制用 **Semaphore(1) + CountDownLatch**：

```text
Semaphore(1) 控制每个工具不并发（同工具只执行一个实例）。
CountDownLatch 控制同层节点全部完成后才能进入下一层。
raceGroup 实现竞速机制：同组节点一个先完成，其他取消。
```

### 第五层：记忆系统

四层记忆架构：

| 记忆层 | 数据结构 | 存储位置 | 用途 |
|---|---|---|---|
| ShortTermMemory | `List<ConversationMessage>` | JVM 内存 + PostgreSQL chat_history | 当前会话上下文 |
| PreferenceMemory | `Map<String, String>` | JVM 内存 + PostgreSQL preferences | 用户结构化画像 |
| LongTermMemory | `List<MemoryItem>` | JVM 内存 + PostgreSQL long_term_memory | 可召回的事实 |
| GraphMemory | Neo4j 节点+边 | JVM 内存 + Neo4j | 记忆之间的关系 |

写入时序：

```text
请求开始时：STM 写入 → saveChatHistory（同步）
回答生成后：STM 写入 → saveChatHistory（同步）
回答生成后：MemoryWriter 异步分类抽取 → 写偏好/长期/图记忆（异步）
写入每 5 条：Consolidation 后台整理（异步）
```

需要强调的重点：

- **短期记忆**是滑动窗口，`maxTurns * 2` 控制大小，`remove(0)` 删除最早的消息。它只保留最近几轮原始聊天。
- **偏好记忆**是轻量级 KV 存储，通过规则匹配（如 `contains("我叫")`）和 LLM 抽取两种方式填充。填充后进入 `memPrefix`。
- **长期记忆**用 `MemoryItem` 存每条事实，包含 `content`（给 LLM）、`embedding`（给召回）、`importance`（给排序和整理）。写入时用 cosine 相似度去重。
- **图记忆**在 Neo4j 里建 `FOLLOWS` 和 `SIMILAR_TO` 边。召回时先用 LTM 找到 seed，再用 Neo4j 扩展 1 跳邻居。

### 第六层：MemoryWriter 和记忆持久化

MemoryWriter 是异步的：

```java
new Thread(() -> writeNow(query, answer), "memory-writer").start();
```

writeNow 内部：

```text
classify(answer) → LLM 分类 JSON → parse → List<Classified>
for each Classified:
  persist(c):
    if identity/preference: 尝试写 PreferenceMemory
    llm.embed(content)
    storeClassified (LTM/GraphMem)
    if added: savePG + syncLastItemPGID
```

**说明它是异步的**——用户拿到回答后，MemoryWriter 才在后台开始跑。不会让用户等。

### 第七层：Consolidation

每写入 `triggerInterval`（默认 5）条新记忆触发一次。

4 个阶段：

```text
① decay importance: 过旧且低重要性的衰减
② pairwise compare: 两两计算 embedding 相似度
③ dedup/merge: 相似的合并，importance 取 max
④ expire: 过旧且低重要性的删除
```

图记忆开启时多一步：**保护入度 >= 3 的节点不被删除**。

### 第八层：降级策略

整个系统强调"降级友好"：

| 故障点 | 降级行为 | 影响范围 |
|---|---|---|
| Planner JSON 解析失败 | 规则规划或降级 chat | 多工具功能不可用 |
| Neo4j 不可用 | 跳过图写入和召回 | 图邻居扩展不可用 |
| MemoryWriter 异常 | 只记 warn | 本条回答无长期记忆 |
| Consolidation 异常 | 只记 warn | 记忆暂时不整理 |
| 数据库不可用 | saveChatHistory 跳过 | 重启无法恢复历史 |
| LLM embedding 失败 | emb = null 继续写入 | 去重/召回可能不准 |


面试时找一张白纸先画这个：

```text

   ▼

   │
   ▼

   │
   ▼
③ buildMemorySystemPrefixWithCtx ──→ memPrefix (system prompt)
   │
   ▼
④ buildHistory ──→ histMsgs (messages)
   │
   ▼
⑤ ChatRouter.decideMode
   │
   ├── chat → LLM.chat
   ├── tool → ToolModeHandler
   ├── react → Planner→GraphRuntime→LLM
   └── rag → RAG recall→LLM
   │
   ▼
⑥ answer
   │
   ▼
⑦ STM.add("assistant", answer) ──→ PostgreSQL
   │
   ▼
⑧ 填充 ChatResponse 元信息
   │
   ▼

   │
   ▼
⑩ 返回 ChatResponse
```

```text
┌─────────────────────────────────────────────┐
│             记忆系统 4 层架构                  │
├─────────┬─────────┬──────────┬──────────────┤
│ STM     │ Pref    │ LTM      │ GraphMemory  │
│ (List)  │ (Map)   │ (Memory  │ (Neo4j 图)  │
│         │         │  Item[]) │              │
├─────────┼─────────┼──────────┼──────────────┤
│ 最近对话 │ K-V     │ 嵌入+    │ FOLLOWS     │
│ 滑动窗口 │ 偏好     │ 语义召回  │ SIMILAR_TO  │
```

### 第 3 张：DAG 执行

```text
Planner 输出:
  n1(get_weather) ──→ n3(合成回答)
  n2(search_web)  ──→ n3(合成回答)

GraphRuntime 分层:
  Layer 0: [n1, n2]  ← 并行执行
  Layer 1: [n3]      ← 等 n1, n2 完成

Semaphore(1): 每个工具限流 1 个实例
CountDownLatch: 等同一层所有节点完成
raceGroup: 同功能工具取最快结果
```

## "如果要改进，你怎么改"

### 常见改进方向

**1. 路由用 LLM 做补充**

当前路由是纯关键词匹配，覆盖有限。可以增加一个 LLM-based router 做 fallback：关键词判断不了时，用轻量 LLM 判断。

**2. 线程管理优化**

MemoryWriter 和 Consolidation 都用 `new Thread`。可以改成线程池 + 消息队列，控制并发上限。

**3. 异步操作的可观测性**

当前 MemoryWriter 和 Consolidation 都是"fire and forget"——失败了只记日志，没有重试、没有监控、没有告警。可以加暴露指标（成功数、失败数、耗时 P99）到 Prometheus/Grafana。

**4. 多用户隔离**

当前 ShortTermMemory 是 Spring 单例，多用户会话会混在一起。可以改成按 sessionId 隔离的 Map。

**5. Consolidation 性能优化**

当前 pairwise compare 是 O(n²)，记忆超过 100 条后性能退化严重。可以加入倒排索引或聚类算法。

**6. 持久化完整性**

stm.add 和 saveChatHistory 没有事务。内存写成功但数据库写失败时，重启后会丢失聊天历史。可以改成先写数据库再写内存，或者用双写确认。

## 面试追问应对策略

### 如果被问到了不会的细节

```text
"这个模块我在开发时主要负责了 X 部分，Y 部分我了解整体设计但没有深入实现细节。根据我对代码的阅读，它的思路是...（说出你的理解）。回去后我可以再看一下具体实现。"

重要的是展现出：你知道这个模块的存在，理解它的设计意图，能说对方向。
```

### 如果被问到"你怎么理解这个系统的局限性"

```text
"当前系统的设计定位是原型/PoC 级别，几个明显的局限性：
第一，单用户隔离不足，ShortTermMemory 是全局单例。
第二，线程管理直接用 new Thread，没有线程池。
第三，无事务保障，写数据库失败可能丢历史。
第四，Consolidation 是 O(n²)，大规模不行。
这些在生产环境都需要优化，但原型阶段追求简单可靠，选择这些取舍是合理的。"
```

### 如果被问到"有没有踩过哪些坑"

```text
"比较典型的坑是 ID 对齐问题。刚开始没有 syncLastItemPGID，内存 ID 和数据库 ID 不一致，导致 consolidate 删除时删错了记录。加了 sync 后才解决。这是一个典型的内存状态和持久化状态不一致的问题。"
```
