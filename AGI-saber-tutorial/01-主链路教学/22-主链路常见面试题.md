# 22 主链路常见面试题

## 面试题 1：一个请求从进到出，经过了哪些步骤？

**完整回答：**

```text
分为 10 个步骤，全部在 UnifiedAgentService.processInternal 里完成：

1. 短期记忆写入：stm.add("user", query)，同时 saveChatHistory("user", query) 保存聊天历史。
2. 异步偏好抽取：runAsyncPreferenceExtraction(query)，启动后台线程用轻量级规则和 LLM 从 query 中提取偏好。
3. 记忆前缀构建：buildMemorySystemPrefixWithCtx(query)，组装 pref.buildContext() + ltm.recall(query) 为 memPrefix。
4. 短期记忆转 histMsgs：ChatHistoryAdapter.buildHistory(stm, query)，把 ConversationMessage 列表转成 LLM 需要的 messages。
5. 路由决策：ChatRouter.decideMode，根据 query 关键词判断走 chat/tool/react/rag。
6. 模式执行：
   - chat：直接调 LLM。
   - tool：ToolModeHandler 三步：选择工具 → getExecute → apply → 结果给 LLM 合成。
   - react：Planner 规划 DAG → GraphRuntime 执行 → LLM 合成。
   - rag：召回知识库文档 → LLM 合成。
7. 助手回答写入：stm.add("assistant", answer)，同时 saveChatHistory。
8. 元信息填充：填充 mode、steps、shortTermCount、longTermCount、preferences。
9. 异步 MemoryWriter：memoryWriter.writeAfterReply(query, answer)，后台线程分类提取长期记忆。
10. 返回 ChatResponse。

整个过程没有用到分布式事务或消息队列，原型阶段以简单为主。
```

**如果面试官追问"有没有哪些步骤是有条件才执行的"：**

```text
有。runAsyncPreferenceExtraction 只在 isRealLLM() 为 true 且抽取出内容时才更新 resp。ChatRouter 根据关键词不同可能跳过或进入不同模式。MemoryWriter 在 !cfg.isRealLLM() 或 answer 为空时直接返回。
```

## 面试题 2：你们怎么判断走 chat/tool/react/rag？

**完整回答：**

```text
ChatRouter.decideMode 使用多层决策：

第一层：explicit 模式（前端显式指定）
  → selectedTools 非空 → react
  → useRag && ragLoaded → rag
  → 否则 → chat

第二层：自动判断
  → needReAct(query) : 4 类子任务关键词计数 >= 2 → react
  → needTool(query) : 6 个关键词包含任一个 → tool
  → needRAG(query, ragLoaded) : 非工具问题且知识库已加载 → rag
  → 兜底 → chat

4 种模式的特点：

| 模式 | 触发条件 | 执行方式 |
|---|---|---|
| chat | 无关键词匹配 | 直接 LLM 对话 |
| tool | 1 类任务关键词 | ToolModeHandler 单工具+LLM 合成 |
| react | >= 2 类任务关键词 | Planner+GraphRuntime+LLM 合成 |
| rag | 非工具且知识库已加载 | 知识库召回+LLM 合成 |

示例：

| query | needTool | needReAct | 最终模式 |
|---|---|---|---|
| "你好" | false | false | chat |
| "上海天气" | true | false | tool |
| "查上海天气并搜索小雨建议" | true | true | react |
```

**如果面试官追问"为什么不用 LLM 做路由"：**

```text
关键词匹配在原型阶段够用：确定性强（同一个 query 永远走同一个模式），速度快（毫秒级），不消耗 token。缺点是覆盖有限——比如"帮我看看要不要带伞"不会触发天气工具。生产环境可以加入 LLM-router 作为补充。
```

## 面试题 3：memPrefix 和 histMsgs 有什么区别？

**完整回答：**

```text
memPrefix 和 histMsgs 都是组装给 LLM 的信息，但来源和用途不同：

| 对比项 | memPrefix | histMsgs |
|---|---|---|
| 内容 | 偏好 + 长期记忆 | 最近几轮原始对话 |
| 来源 | PreferenceMemory + LongTermMemory.recall | ShortTermMemory |
| 放哪里 | system prompt | messages |
| 语义角色 | "你该知道的用户背景" | "刚刚聊了什么" |
| 是否跨会话 | 是 | 否 |
| 生成方式 | buildMemorySystemPrefixWithCtx | ChatHistoryAdapter.buildHistory |

例子：

```text
system prompt:
  【用户偏好】
  姓名: 小李              ← memPrefix

  你是一个简洁的AI助手。   ← basePrompt

messages:
  user: "我叫什么？"       ← histMsgs
```

两者同时发给 LLM，但语义角色不同。memPrefix 告诉模型"用户是谁、喜欢什么"，histMsgs 告诉模型"当前对话进行了什么"。
```

## 面试题 4：单工具模式和 ReAct 模式有什么区别？

**完整回答：**

```text
| 对比项 | ToolModeHandler (tool) | ReAct (react) |
|---|---|---|
| 触发条件 | 1 类任务关键词 | >= 2 类任务关键词 |
| 工具数量 | 通常 1 个 | 多个 |
| 是否规划 | 不需规划 | Planner 规划 DAG |
| 执行方式 | 顺序执行 | 并发执行（拓扑分层） |
| 竞速机制 | 无 | 有 raceGroup |
| LLM 调用次数 | 2 次（选工具+合成） | 3+ 次（规划+执行+合成） |
| 中间事件 | 有（nodeStart/nodeDone） | 有（含 graphReady） |
| SSE 事件 | 无 graphReady | 有 graphReady |

ToolModeHandler 的 5 步流程：

```text
① selectTools(query, ts) → 选择最匹配的工具
② tool.getExecute() → 拿到 ExecuteFunction
③ execute.apply(args) → 调用执行
④ 结果 → LLM 合成回答
⑤ 保存 steps 到 resp
```

ReAct 的流程：

```text
① Planner.planGraph → LLM 规划 DAG
② GraphRuntime.execute → 拓扑排序+并发执行
③ LLM synthesis → 合成最终回答
```

核心区别：tool 模式是"先调工具再总结"，react 模式是"先规划再并发执行再总结"。
```

## 面试题 5：Planner 怎么规划任务？

**完整回答：**

```text
Planner.planGraph 用 LLM 把用户问题分解为 DAG。

步骤：

① 构建 LLM 请求：
   系统 prompt 包含任务规划指令。
   用户 prompt 包含 query + 可用工具列表 + 当前历史。
   LLM 被要求输出 JSON 格式：nodes 数组 + edges 数组。

② JSON 解析（3 层降级）：
   第 1 层：直接解析 LLM 原始 JSON。
   第 2 层：清洗后解析（去掉 ```json 头/尾，去掉 Markdown）。
   第 3 层：规则规划（每个工具一个独立节点，无依赖关系）。
   全部失败 → 降级到 chat 模式。

③ 构建 Graph 对象：
   nodes = [Node{id, tool, args, dependsOn}]
   edges = [Edge{from, to}]

④ 返回 Graph 给主链路。

LLM 输出示例：

```json
{
  "nodes": [
    {"id":"n1","tool":"get_weather","args":{"city":"上海"}},
    {"id":"n2","tool":"search_web","args":{"query":"上海 小雨 建议"}}
  ],
  "edges": []
}
```

关键设计点：

| 设计 | 说明 |
|---|---|
| 3 层降级 | LLM 可能输出非法 JSON，逐层降级保证功能不中断 |
| edges 定义依赖 | 空 edges = 全并行，有 edges = 分层执行 |
| 每个节点独立参数 | 参数在规划时由 LLM 生成，运行时不变 |
```

## 面试题 6：GraphRuntime 怎么控制并发？

**完整回答：**

```text
GraphRuntime 用拓扑排序 + Semaphore + CountDownLatch 控制并发执行。

步骤：

① 拓扑排序：
   根据 edges 把 nodes 分成若干层。
   每层内的节点没有依赖关系，可以并行。
   层间有依赖关系，必须前一层全部完成才能执行下一层。

② 并发控制：
   每个工具节点初始化时创建 Semaphore(1)。
   用 Semaphore 限制每个工具同时只执行一个实例（防止同工具并发）。
   用 CountDownLatch 等待同层所有节点完成。

③ 竞速机制（raceGroup）：
   同一 raceGroup 的节点中只要有一个先完成，就取消同组其他执行。
   用于：同一个查询可以用不同 API 实现，取最快的结果。

④ 执行流程：

```text
for each 拓扑层:
  for each 节点 in 层:
    启动线程执行
    发送 SSE: nodeStart
    Semaphore.acquire()
    try { result = tool.execute.apply(args); }
    catch { result = error; }
    finally { Semaphore.release(); }
    发送 SSE: nodeDone
    发送 SSE: observation
    检查 raceGroup（如果有）
  CountDownLatch.await() ← 等待本层所有节点完成
```

**如果面试官追问"为什么用 Semaphore(1) 而不是直接并发"：**

```text
同一个工具可能被多次调用（比如两次 get_weather 查不同城市），用 Semaphore(1) 保证同一个工具不会同时执行两次，避免资源竞争。
```

## 面试题 7：记忆系统什么时候写入？

**完整回答：**

```text
记忆写入分三个时机：

| 时机 | 写入内容 | 同步/异步 | 操作 |
|---|---|---|---|
| 请求开始时 | 短期记忆 | 同步 | stm.add("user", query), saveChatHistory |
| 请求开始时 | 偏好记忆 | 异步 | runAsyncPreferenceExtraction |
| 回答生成后 | 短期记忆 | 同步 | stm.add("assistant", answer), saveChatHistory |
| 回答生成后 | 长期记忆 | 异步 | MemoryWriter.writeAfterReply |
| storeClassified 触发 | 长期记忆 | 同步 | ltm.storeClassified / graphMem.storeClassified |
| storeCount >= threshold | Consolidation | 异步 | ltm.consolidate / graphAwareConsolidate |

短期记忆写入（同步）：
```text
请求开始：stm.add("user", query) → 当前问题
请求结束：stm.add("assistant", answer) → 当前回答
```

偏好记忆写入（异步）：
```text
请求中间：extractAndSave(query) → 轻量级规则 + LLM 提取
```

长期记忆写入（异步）：
```text
请求结束：MemoryWriter.classify(answer) → LLM 分类 → storeClassified → PG
```

注意：短期记忆写入是同步的，长期记忆写入是异步的。因为前者影响当前回答，后者不需要阻塞用户。
```

## 面试题 8：ID 同步是什么意思？

**完整回答：**

```text
ID 同步解决内存 ID 和数据库 ID 不一致的问题。

写入过程：

```text
nextId=0 (内存计数器)
  → new MemoryItem(nextId++, ...) → id=0, nextId=1
  → items.add(item) → items 最后一条 id=0
  → infra.saveLongTermItemClassified → PG 返回 pgId=135
  → syncLastItemPGID(135):
       items.get(size-1).id = 0 → 135
       nextId = 136
```

为什么需要？

```text
内存 ID 从 0 开始自增。
PG 的 SERIAL ID 可能从任意值开始（比如之前已有记录，从 135 开始）。

不对齐时：
  consolidate 想删除 id=0 → DELETE WHERE id=0 → PG 里没有 id=0
  这条记忆永远留在 PG 里了。这是僵尸数据。
```

图记忆的额外同步：

```text
graphMem.syncLastItemPGID(pgId) 除了对齐内存和 PG，还会：
  → 更新 Neo4j 节点的 mem_id 属性
  → 保证三个存储层 ID 一致
```

## 面试题 9：如果 Neo4j 挂了，系统还能工作吗？

**完整回答：**

```text
能。这是系统的优雅降级设计。

Neo4j 挂了时：

| 功能 | 行为 | 影响 |
|---|---|---|
| LTM 写入 | 正常写入 LongTermMemory | 无影响 |
| GraphMemory 写入 | 写 LTM，跳过 Neo4j | 图边不持久化 |
| LTM 召回 | 正常召回 | 无影响 |
| 图邻居扩展 | 不执行（没有 Neo4j 可用） | 召回不扩展邻居 |
| Consolidation | 走 ltm.consolidate 不走 graphAware | 无图节点保护 |

代码层面：

```java
// MemoryWriter.persist 里
if (graphMem != null) {
    added = graphMem.storeClassified(...).added();
    // graphMem.storeClassified 内部：
    //   ① 写 LTM（正常）
    //   ② try { 写 Neo4j } catch { log.warn } ← 优雅降级
    //   ③ 返回 added 状态
}
```

所以 Neo4j 挂了时：

```text
长期记忆仍然可以写入。
长期记忆仍然可以召回。
只是图关系不持久化，图邻居扩展不可用。
系统仍然可以正常工作，只是"降级"为纯 LTM 模式。
```

## 面试题 10：这个系统的最大设计亮点是什么？

**完整回答：**

```text
四个设计亮点：

**1. 4 层记忆架构**
ShortTermMemory 负责当前会话上下文，PreferenceMemory 负责结构化用户画像，LongTermMemory 负责语义可召回的事实向量，GraphMemory 负责记忆之间的关系网络。四层各司其职，互不干扰。短期记忆是滑动窗口，长期记忆是持久化向量存储，偏好是轻量级 KV，图记忆是知识图谱。

**2. 4 模式路由决策**
ChatRouter 用关键词匹配决定 chat/tool/react/rag。ChatRouter 纯关键词判断快且稳定（毫秒级），优先 explicit 模式，再依次判断 needReAct/needTool/needRAG，兜底 chat。不需要 LLM 参与路由决策，节省 token。

**3. DAG 并发执行**
Planner+GraphRuntime 支持多工具规划为 DAG，拓扑排序分层后并行执行，支持竞速机制（raceGroup）。同一个 DAG 内的无依赖节点同时跑，有依赖的分层跑，竞速组淘汰慢的。LLM 规划+运行时并发+竞速淘汰，三者配合。

**4. 全链路降级**
每个模块都有降级策略：Planner 3 层 JSON 降级，GraphRuntime 异常只影响单节点，Neo4j 挂了不影响 LTM，Consolidation 失败只记 warn，MemoryWriter 写失败不影响主响应。没有单点故障——任何模块出问题，系统都不会完全不可用。
```

## 快速记忆卡

```text
10 步流程：
STM写 → 偏好抽 → 前缀建 → 历史转 → 路由判 → 模式执 → 回答存 → 元信息填 → 异步写 → 响应返

4 层记忆：
STM(窗口) → Pref(KV) → LTM(向量) → Graph(图)

4 种模式：
chat → tool → react → rag

3 个异步：
偏好抽取 → MemoryWriter → Consolidation

2 个同步写：
stm.add + saveChatHistory

1 个目的：
给 LLM 提供最佳上下文，最小化用户等待
```
