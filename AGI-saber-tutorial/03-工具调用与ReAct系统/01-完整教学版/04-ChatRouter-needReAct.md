# 04-ChatRouter-needReAct

## 1. 这个方法解决什么问题

有些问题不是一个工具能解决的。比如：

```text
"查一下上海天气，并搜索小雨出门建议"
```

这需要两个步骤：先查天气，再搜索建议。这就是 `needReAct` 要识别的情况——**需要多步骤推理的 query**。

## 2. 方法源码，源码里加注释

```java
/**
 * 位置：ChatRouter.java:34-42
 *
 * 参数：query - 用户原始输入
 * 返回：true = 需要多步推理（ReAct），false = 单工具足以
 */
public static boolean needReAct(String query) {
    String q = query == null ? "" : query.toLowerCase(Locale.ROOT);  // ① 转小写
    int count = 0;
    if (q.contains("时间") || q.contains("几点")) count++;            // ② 时间类
    if (q.contains("天气")) count++;                                  // ③ 天气类
    if (q.contains("总结") || q.contains("汇总")) count++;            // ④ 综合类
    if (q.contains("查") || q.contains("搜索")) count++;              // ⑤ 搜索类
    return count >= 2;                                                // ⑥ 关键：>= 2
}
```
### 2.1 逐行解释

下面按照源码里的编号，把这个方法的每一步拆开说明。重点看每一步改变了什么数据、影响了哪个后续方法。

**① 转小写**

转小写。这一步做数据形态转换，让下游方法拿到它期望的结构。

**② 时间类**

时间类。命中这类关键词说明 query 可能需要时间工具；这里只做意图计数或布尔判断，不执行工具。

**③ 天气类**

天气类。命中天气关键词说明 query 可能需要天气工具；具体城市参数会在 ToolService.decide 或 PreferenceFiller.fill 中处理。

**④ 综合类**

综合类。这是 ReAct 触发的核心：多个语义维度同时出现时，单工具模式不够，需要 Planner 生成多节点任务图。

**⑤ 搜索类**

搜索类。命中 `查` 或 `搜索` 说明 query 可能不止一个信息源，和天气、时间等维度叠加后就会触发 ReAct。

**⑥ 关键：>= 2**

关键：>= 2。这是 ReAct 触发的核心：多个语义维度同时出现时，单工具模式不够，需要 Planner 生成多节点任务图。

## 4. 返回值/副作用解释

**返回值**：`boolean`

**副作用**：无。纯函数。

**核心逻辑：`count >= 2`**

四个维度各算 1 分，不重复累计。同一维度内是 OR 关系（`"查" OR "搜索"` 只加 1 分，不会加 2 分）。

| 维度 | 关键词 | 命中条件 |
|---|---|---|
| 时间 | `几点` OR `时间` | 1 分 |
| 天气 | `天气` | 1 分 |
| 综合 | `总结` OR `汇总` | 1 分 |
| 搜索 | `查` OR `搜索` | 1 分 |

**四个维度的语义**：

- **时间 + 天气** → "现在几点？天气怎么样？"（两个独立工具）
- **天气 + 搜索** → "上海天气怎么样？搜索出门建议"（天气结果可能影响搜索方向）
- **时间 + 搜索** → "现在几点了？帮我查一下还有没有高铁"（真实场景较少，但规则覆盖）
- **天气 + 综合** → "总结一下本周天气"（可能多天天气查询 + 汇总）

## 5. 这一步在完整链路中的位置

```text
decideMode(query, ...)
  ├── explicit? → 跳过
  ├── needReAct(query)? → react    ← 你在这里
  ├── needTool(query)?
  ├── needRAG(query)?
  └── chat
```

**`needReAct` 是自动模式下的第一个判断。** 如果它返回 `true`，`needTool` 和 `needRAG` 都不会被检查。

这与设计意图一致：ReAct 是"超集模式"——一个 ReAct 图可以包含单工具节点，但单工具模式包含不了多步推理。所以 ReAct 优先。

## 6. 用两个例子跑一遍

**例子 A：单工具触发（count=1）**

```text
query = "上海天气怎么样？"

时间维度：无"几点"/"时间" → count=0
天气维度：有"天气"       → count=1
综合维度：无"总结"/"汇总" → count=1
搜索维度：无"查"/"搜索"   → count=1

count=1 < 2 → 返回 false
→ decideMode 继续检查 needTool → 返回 "tool"
```

**例子 B：多步触发（count>=2）**

```text
query = "查一下上海天气，并搜索小雨出门建议"

时间维度：无                    → count=0
天气维度：有"天气"              → count=1
综合维度：无                    → count=1
搜索维度：有"查"+"搜索"         → count=2

count=2 >= 2 → 返回 true
→ decideMode 直接返回 "react"
```

## 7. 常见误解

**误解一："count >= 2 表示需要 2 个或更多工具"**

不是。`count >= 2` 意味着 query 覆盖了两个**语义维度**，不是两个工具。比如 `"现在几点？天气怎么样？"` 覆盖时间 + 天气 = count=2，但实际可能需要两个工具（`get_time` + `get_weather`）。

**误解二："同一个维度出现两次关键词会加 2 分"**

不会。`"查一下天气并搜索建议"` 中"查"和"搜索"都在搜索维度，只加 1 分。加上天气维度才到 2 分。

**误解三："needReAct 等于 NLP 意图识别"**

远不是。`count >= 2` 是极其粗糙的启发式规则。它的局限很明显：

```text
"明天上海天气怎么样？"            → count=1 → tool（合理）
"明天上海天气怎么样，会比今天热吗？" → count=1 → tool（不合理，需要比较，应该是 react）
"北京和上海天气对比"              → count=1 → tool（不合理，需要两次查询，应该是 react）
```

真正的意图识别应该由 LLM 完成（Planner 会做这件事）。`needReAct` 只是路由层的快速预判——用极低成本过滤掉显而易见的单工具请求，避免每条 query 都走 LLM 规划。

**误解四："needReAct 可以替代 Planner"**

不能。`needReAct` 只决定**要不要**进入 ReAct 模式。进入之后，具体怎么拆节点、节点间的依赖关系、参数的填充，全部由 `Planner.planGraph`（第 18 章）完成。
