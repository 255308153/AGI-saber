# 34-面试总结

## 1. 按方法链叙述，不按概念清单

V3 的面试总结按"概念是什么"组织（Tool 是什么、ToolParam 是什么……）。V4 按"通过哪个方法实现"组织——面试时你沿着真实调用链往下说，每一步有方法名、有参数、有返回值、有实例。

## 2. 一分钟版：两条链

```text
单工具链（上海天气怎么样？）：
  processInternal → decideMode("tool") → ToolModeHandler.run
    → decide("get_weather") → fill(不覆盖显式参数) → execute.apply
    → llm.chat 总结

ReAct 链（查天气 + 搜索建议）：
  processInternal → decideMode("react") → ReActLoop.runStream
    → planGraph(LLM规划3节点) → TaskGraph(建图) → GraphRuntime.execute(拓扑并行+竞速)
    → ChatGenerator.generate(综合observations)
```

## 3. 三分钟版：关键设计决策

### 3.1 为什么工具有定义和执行两面？

```text
Tool = 定义（name + description + parameters + execute）
ToolParam = 参数 schema（name + type + description + required）
ToolCallResult = 一次调用记录（toolName + params + toolResult）

区分的原因：
- Tool 是全局单例，被所有请求共享 → 不能存调用状态
- ToolCallResult 是每次请求独立的 → 存决策 + 参数 + 结果
- ToolParam 是 LLM Planner 需要的说明 → LLM 据此判断传什么参数
```

详见第 11 章。

### 3.2 为什么需要 PreferenceFiller？

```text
问题：用户说"今天天气怎么样？"——没提城市
方案：从偏好记忆读"城市=上海"，补到 params 里

关键约束：不覆盖用户显式输入
  "上海天气怎么样？" → params 已有 city="上海" → fill 跳过
  "今天天气怎么样？" → params 无 city → fill 写入 "上海"

实现：PREF_TO_PARAM 映射表 + 双循环遍历 + v==null||empty 检查
```

详见第 14 章。

### 3.3 为什么单工具不够用？

```text
"查一下上海天气，并搜索小雨出门建议" 需要两个工具

ToolModeHandler 只能决策并执行一个工具 → 不够
ReAct 模式：Planner 拆成多个 Node → TaskGraph 表达依赖 → GraphRuntime 按拓扑层并行执行

关键差异：
- 单工具：规则决策（String.contains），顺序执行
- ReAct：LLM 规划（prompt → JSON），DAG 并行 + 竞速
```

详见第 13, 17, 18 章。

### 3.4 图执行为什么需要拓扑排序？

```text
n2 依赖 n1 的输出 → n2 必须在 n1 之后执行
但 n2 和 n3 无依赖 → 可以并行

拓扑分层（Kahn 算法）：
  L0: [n1]             ← 先执行
  L1: [n2, n3]         ← n1 完成后两者并行

如果简单 for 循环：n1 → n2 → n3 (串行)
拓扑分层后：耗时 = n1 + max(n2, n3) (并行)
```

详见第 23, 24 章。

### 3.5 竞速（raceGroup）解决什么问题？

```text
search_web（互联网搜索）和 rag_search（私人知识库搜索）
→ 功能等价，谁先返回用谁的

runRace 实现 first-success-wins：
- 并发启动两个线程
- 第一个成功者的结果写入 observations
- 另一个线程被 interrupt，节点标记 SKIPPED
```

详见第 27 章。

### 3.6 tool.getExecute().apply(params) 为什么参数是 Map？

```text
Function<Map<String, Object>, String>

Map 而非固定类 → 不同工具有不同参数，不需要为每个工具定义子类
String 返回值 → 所有工具统一输出类型，LLM 统一处理
Object value → 可以传 String/Integer/Double，但目前只用到 String
```

详见第 15 章。

## 5. 核心区分对比

| 对比项 | 第一个 | 第二个 | 关键区别 |
|---|---|---|---|
| 数据类 | `Tool`（定义） | `ToolCallResult`（记录） | 一个是说明书，一个是通话记录 |
| 参数 | `ToolParam`（schema） | `params`（值） | 一个是"需要什么"，一个是"传了什么" |
| 模式 | `tool` | `react` | 单工具顺序 vs 多工具 DAG |
| 决策 | `decide`（规则） | `planGraph`（LLM） | 6个关键词 vs prompt→JSON |
| 执行 | `runSingle` | `runRace` | 各自独立 vs first-success-wins |
| 总结 | 内联 `llm.chat` | `ChatGenerator.generate` | 单工具的 userMsg vs 多 observation 的 genPrompt |
| 参数补全 | `PreferenceFiller.fill` | 无（LLM 直接填） | 规则补参 vs LLM 推理确定 |
| 并发控制 | 无 | Semaphore(maxParallel) | 单工具不需要 vs 防止打爆 API |
| 状态管理 | `ToolCallResult.toolResult` | `Node.status` + `Node.result` | 一个字段 vs 状态机 |
